1 Hintergrund

Es geht darum, IPEmotion so anzupassen, dass ein Mehrwert für die Prüfstandsanwender der IPEtec Prüfstände gegeben ist.

Aktuell ist IPEmotion fast ausschließlich auf den Automotiv Bereich zugeschnitten. Es existieren nur rudimentäre Funktionen

für Prüfstandsanwendungen. Aussage der Kunden „IPEmotion ist für Prüfstandsanwendung nicht intuitiv zu benutzen“.

Die Firma Mahle hat bereits einen Prüfstand für Mahle Suzhou, China beauftragt. Ein weitere Prüfstand in Armherst, USA ist

bereits in der Verhandlungsphase. Mahle Luxembourg will in absehbarere Zeit alle Prüfstände durch IPEtec Prüfstände

erneuern. Aktuell wird eine proprietäre Software (von Delphi) zum Betreiben der Prüfstände genutzt. Die Software ist bereits
abgekündigt. Mahle (Allen Dunn) macht die Beschaffung weitere Prüfstände davon abhängig, dass die Grundfunktionalität in

IPEmotion komfortable umgetzt wird. Das hat ausserdem den Nutzen, dass auch alle andere IPEtec Kunden davon
profitieren. Dieses Dokument hat das Ziel IPEmotion Anpassungen zu beschreiben, dass nicht nur Mahle, sondern alle davon

Nutzen tragen -> Steigerung der Attraktivität unserer Prüfstände durch komfortable Bedienung.

Die Anforderungen wurden durch NF, SHA, MA und MD erstellt.

1.1 Betriebspunkte einer x-Punkte Matrix

Die häufigste Anwendung eines Prüfstands besteht darin, eine Matrix mit mehreren Messpunkten „abzufahren“. Diese wird

meist durch Normen oder Kunden-interne Dokumente vorgegeben. Eine entsprechende Beispiel-Matrix kann von IPEtec /

IPEengineering bereitgestellt werden. Diese Matrix soll relativ einfach eingegeben und später vom Prüfstand (IPEmotion)

umgesetzt werden.

Innerhalb dieser Matrix gibt es Blocks bzw. Cycles (Kundenbezeichnung). Innerhalb der Blocks kann es mehrere

unterschiedliche Steps geben. Jeder Step beinhaltet einen unterschiedlichen Betriebspunkt. Dieser Punkt wird auf bestimmte
Art und Weise angefahren und für eine vorgegebene Dauer gehalten, bevor es zum nächsten Step geht. Manchmal gibt es

nur einen Step in einem Block. Mehrere Blöcke werden dann zu einer Sequence Control zusammengeführt.

Jeder Block/Cycle hat einen Start-, einen BreakIn-, viele Test-Cycles und einen Check-Cycle. Der Start-Cycle definiert, wie

der Prüfstand beim Start „eingefahren“ wird, bevor der eigentlich Test-Cycle beginnt. Er wird nur einmalig vor jedem Lauf

ausgeführt. Der Test-Cycle stellt dem eigentlichen Test des Prüflings dar. Dieser Test-Cycle kann beliebig oft wiederholt

werden. Wie o.a. enthält der Test-Cycle unterschiedliche Steps. Ein BreakIn-Cycle definiert die Wiederaufnahme des Test-
Cycles, wenn der Test (aus welchen Gründen auch immer) unterbrochen wurde. Ein manuelles Unterbrechen als auch ein

automatisches Abschalten ist dabei egal. Mit dem Break-In Cycle wird sichergestellt, dass nach der Unterbrechung, egal zu

welchem Zeitpunkt der Test-Cycle unterbrochen wurde, der letzte Punkt des Test-Cycle wieder voll-automatisch erreicht /

aufgenommen wird. Das kann dabei über einen Start-Cycle erreicht werden. Der Check-Cycle überprüft, ob gemessene

Werte soweit in Ordnung sind (Überprüfung Soll/Ist-Werte).

2.2 Verknüpfung mit Profil-Generator

IM-23497 - Verknüpfung mit Profil-Generator

Anforderung: Im "Test Bench View" soll die Möglichkeit gegeben sein, den Profilgenerator zu starten.

Neben dem jeweilig einzutragenden Wert im dem Grid aus Item 2.1 kann über einen Button (siehe Beispiel oben) der

Profilgenerator gestartet werden. Wenn über den Profilgenerator anstelle eines fixen Wertes ein Profil (ein Verlauf)

abgefahren wird, so soll das Eingabefeld mit anderem farblichem Hintergrund dargestellt werden.

Es soll die Möglichkeit gegeben sein, zw. Profil und Fixwert zu wählen.

Wenn ein Profil gewählt wurde, sollte der letzte aktuelle Wert zu dem jeweiligen Zeitpunkt dargestellt werden.

Beispiel: Profil wird wiederholend ausgeführt (Loop aktiv). Die Duration von Sequence Block = 20 Minuten.

--> Nach Ablauf von 20 Minuten steht Wert bei 2x7 + 6 = 20 Minuten. Rückgabe Wert zum Zeitpunkt 20 Minuten = 120 bar

(Berechnet).

2.3 Check Block

IM-23498 - Check Block

Anforderung: Für jede Zeile in dem "Test Bench View" Grid kann unten im Eigenschaftsfenster ein "Check block" ausgewählt

werden. Der Check block wird wie jeder andere Sequence Block definiert.

Sobald aber der Check block ausgewählt wird, soll automatisch die Eigenschaft "Customer step" für diesen Block und die

enthaltenen Schritte deaktiviert sein.
